## 上下文

### 业务目标
青沐生命科技「青沐」微信小程序商城接入微信支付 V3，让用户能真实购买商城商品（当前 V1.0 仅支持积分全额兑换）。

### MVP 范围（5 件核心）
1. 后端 `wxpay` module：unifiedOrder + 验签 + AES 解密
2. `orderService.create` 改造：payAmount>0 时调微信统一下单
3. 支付回调路由：验签解密 → Order.status=paid + 写 WalletTransaction + 幂等
4. 前端 `wx.requestPayment` 接入 order-confirm
5. 单测 + 沙箱 e2e 验证

- 状态机校验（任意状态→任意状态跳的硬性校验）
- 积分抵扣比例从 DB 读（仍硬编码 POINTS_TO_YUAN=0.01）

## 详细步骤

### Task #51：后端 wxpay module 骨架

**新文件**：
- `apps/server/src/modules/wxpay/wxpay.service.ts`
  - `unifiedOrder(input: { orderId, openid, description, totalFen, outTradeNo })` → `{ prepayId, nonceStr, timestamp, sign }`
  - `verifyAndDecryptNotify(rawBody, headers)` → `{ outTradeNo, transactionId, amount, success, decrypted: object }`
  - `generateAuthorization(method, urlPath, body)` → string 内部工具
  - `aesGcmDecrypt(ciphertext, key, nonce, aad)` → string
- `apps/server/src/modules/wxpay/wxpay.routes.ts`
  - `POST /api/wxpay/notify` 公开路由（`config: { public: true }`） — 微信回调
  - 留 `POST /api/wxpay/refund` 占位（scheme + 403）— MVP 不实现
- `apps/server/src/modules/wxpay/wxpay.schema.ts`
  - `WxpayNotifyResourceSchema`（解密后 resource 字段校验）
  - `WxpayNotifyDecryptedSchema`（含 out_trade_no / transaction_id / amount / success / openid）

**修改**：
- `apps/server/src/config/env.ts` — 加 `WX_MCH_ID` / `WX_PAY_KEY` / `WX_NOTIFY_URL` / `WX_MCH_SERIAL_NO` / `WX_MCH_PRIVATE_KEY_PATH` / `WX_PLAT_CERT_PATH` 字段
- `apps/server/src/app.ts` — 注册 wxpay 路由模块

**关键技术**：
- 签名用 Node `crypto`：`crypto.sign('RSA-SHA256', message, privateKey)` + `base64`
- AES-GCM 解密：`crypto.createDecipheriv('aes-256-gcm', key, iv).setAuthTag(tag)`
- 平台证书：本地读 `WX_PLAT_CERT_PATH`（开发期放测试证书，生产期从 `certificates` API 拉）

**预期产出**：wxpay module 独立编译通过，env.ts 通过 Zod 校验。

### Task #52：orderService.create 调统一下单

**修改**：
- `apps/server/src/modules/mall/order.service.ts` `create()` 方法：
  - 当 `payment=ON` 且 `payAmount > 0`：
    1. 事务内先建 Order(status=pending_pay, payChannel='wxpay')
    2. **事务外** 调 `wxpay.unifiedOrder({ outTradeNo: order.id, ... })`
    3. 返回 `{ orderId, prepayId, payParams }` 给前端
- `apps/server/prisma/schema.prisma` Order model 加 `payChannel String?` + `prepayId String?` + `wxTransactionId String?`（idempotent 幂等 key）
  - 走迁移：`pnpm prisma migrate dev --name add_pay_fields_to_order`

**注意**：
- 微信 API 在 `prisma.$transaction` 之外调（外部 IO 不能在事务内）
- 积分扣减仍走事务（payAmount=0 的情况不变）
- payAmount=0 的下单流程不变（直接走积分全额兑换 status=paid）

**预期产出**：create 返回 schema 扩展含 prepayId，前端可发起支付。

### Task #53：支付回调路由 + 状态推进

**修改**：
- `apps/server/src/modules/wxpay/wxpay.routes.ts` `POST /api/wxpay/notify`：
  1. 读 raw body + 头（验签必走原始 body，不可 JSON 解析后再验）
  2. 调 `verifyAndDecryptNotify` 验签 + 解密
  3. 查 Order.by(outTradeNo)
  4. **幂等**：若 Order.wxTransactionId === 解密的 transactionId 直接返回 200（防重）
  5. 事务内：
     - Order.status='paid' + wxTransactionId + paidAt
     - 写 WalletTransaction(type=recharge/refund, amount=payAmount, wxTransactionId=transactionId)
     - 若 payAmount>0：Wallet.balance += payAmount（不！是消费不是充值）
     - 修正：Order.payAmount 是用户**支付**的金额 → 落到 wallet 是 **商家** 的钱；用户不增加 wallet 余额
     - 实际：用户 wallet 余额不变（当前架构下用户用积分不用余额），但需记录交易流水以备对账
  6. 返回 200 让微信停止重试

**注意**：
- 路由必须用 `config: { public: true }`（不走 JWT 校验）
- raw body 在 Fastify 默认会被 JSON parse，需注册 `addContentTypeParser('application/json', { parseAs: 'string' })` 保留原 body
- 验签失败必须返回 4xx/5xx 让微信重试

**预期产出**：fastify inject 测试，回调 → Order.status=paid + WalletTransaction 记录。

## 关键技术决策

| 决策 | 理由 |
| --- | --- |
| **Node `crypto` 自研签名/加密** | 不引外部 SDK，减少依赖；V3 协议其实就 RSA 签名 + AES-GCM |
| **回调路由 raw body 解析** | 验签必须用原 body 字节；Fastify 默认 JSON parse 会破坏 |
| **微信 API 在事务外调** | 外部 IO 不能在 DB 事务内（事务超时/回滚 vs IO 已发送） |
| **`payChannel`+`wxTransactionId` 索引** | 幂等查 + admin 按渠道查订单 |
| **不引 `wechatpay-apiv3-nodejs` 等 SDK** | 包都老旧（>=2 年没更新），自研可控；代码量不大 |

## 风险与对策

| 风险 | 对策 |
| --- | --- |
| APIv3 协议细节踩坑（头部 / 签名 / 加密） | 写完先跑通 curl 命令 + 用沙箱验收用例（参考官方支付验收指引） |
| 回调验签字符串构造错误 | 严格按规范"HTTP_METHOD\nURL_PATH\nTIMESTAMP\nNONCE_STR\nBODY\n" |
| 事务边界错误 | 微信 API 调放在 `$transaction` 之前；DB 写入仍走事务 |
| 幂等失效（重发回调） | 数据库加 unique 索引 `@@unique([wxTransactionId])` 防重复 |
| 沙箱测试数据残留 | e2e 用 prefix `e2e-wxpay-` + afterAll 强清 |
| 私钥文件泄露 | 加 `.gitignore` + 文档化部署时由运维注入 |
