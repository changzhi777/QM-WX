## 1. 背景与目标

沐禾健康以**微信小程序**为主端，需接入多品牌智能手表运动健康数据（跑步/心率/睡眠/步数），覆盖佳明/华为/高驰 COROS/松拓/小米/vivo/OPPO 等主流品牌。

**核心矛盾**：
- 微信小程序无法嵌入原生 SDK（Apple HealthKit / Android Health Connect）
- Health Connect 依赖 Google 框架，国行手机（华为/小米/vivo/OPPO）基本不可用
- 各品牌官方 API 开放程度不一（华为面向鸿蒙应用 / 小米生态链企业制 / vivo OPPO 无公开入口）

**目标**：分阶段接入，优先高频品牌（佳明/华为/COROS），无 API 品牌走兜底（文件导入/截图识别）。

## 4. 技术架构（开发任务细化）

### 4.1 dataSource 扩展

`Checkin.dataSource` 现有：`manual / garmin / coros / huawei_export / sport_screenshot / apple_health`。

新增：
- `huawei_kit`（阶段 2.6 华为官方）
- `terra_garmin / terra_coros / terra_suunto / terra_zepp / terra_strava`（阶段 3.3 Terra 聚合）

映射到 `device-parser.registry` + `Checkin` 表，幂等去重 key = `userId + date + distanceKm + dataSource`（复用 V0.2.60）。

### 4.2 聚合平台 webhook 接入（阶段 3.3）

```
Terra/Junction webhook → POST /api/device/terra-webhook (public，签名校验)
  → 数据标准化映射（activity → distanceKm/durationSec/heartRate/pace）
  → sportService.checkin（dataSource=terra_xxx，幂等去重）
  → 睡眠/心率/血氧落 HealthRecord 表（复用 V0.1.43 范式）
```

复用：`upload-parse.job` 异步解析范式 + `interpretScreenshot` 标准化映射 + V0.2.60 去重。

### 4.3 前端 H5 授权（阶段 3.4）

复用 V0.2.63 H5 fallback 范式：
- 小程序内引导 → `issueH5Token`（Redis 5min token）
- H5 页完成 Terra OAuth 授权 → 回调绑凭证
- 回小程序查看同步状态 + 历史数据

### 4.4 华为鸿蒙端（阶段 2.6，需鸿蒙开发）

鸿蒙应用/元服务采集运动健康 → POST qingmulife.cn/api/device（dataSource=`huawei_kit`）→ 落 Checkin。

> 注：鸿蒙端开发需 HarmonyOS 工程师（非小程序/后端范畴），本仓库只承接后端接收 + 数据展示。

## 5. 排期与关键路径

### 关键路径（瓶颈）
1. **0.4 合规出境**（2-4w，法务，硬前置）—— 与阶段1并行启动
2. **2.5 华为测试权限审核**（5-15 工作日）+ **2.8 商用审核**（7-15 工作日）—— 华为审核是阶段2最长路径
3. **3.2 Garmin 开发者计划**（2-4w）—— 阶段3 并行瓶颈

### 并行机会
- 阶段 0.4（合规）与阶段 1（验证）并行
- 阶段 2（华为）与阶段 3（聚合）可并行（不同团队/审核流）
- 阶段 3.2（Garmin 申请）与 3.3（后端 webhook 开发）并行

### 估算总工期（乐观→悲观）
- 决策+验证：1-2w（阶段0+1）
- 华为全链路：6-10w（阶段2，审核驱动）
- 聚合平台：4-6w（阶段3）
- 上线验收：1-2w（阶段5）
- **总计：~12-20w**（华为审核 + 合规出境是主要不确定性）
